# 论证层：血肉审计

骨架立住之后，看每一节的肉是不是真的。论证层的病是**无本之木**：判断句悬空、因果跳步、类比装饰、细节全是"任何人都写得出"的通货。

面向技术传播的定位：这里的"据"不是学术引用，而是机制解释、实测细节、算账和真实的认知轨迹。代码是证据的一种，但不是主要的一种。

## 挣来的具体性

最重要的概念。一句话是否有分量，取决于它是不是**只有真做过的人才写得出**。

| 通货（谁都能写） | 挣来的（做过才知道） |
|------|------|
| 国内信用卡可能无法支付 | 招行 Visa 多次尝试全部失败，换美国节点、英国节点照样不行 |
| 保号成本很低 | £15 余额 ÷ £0.6/年 = 25 年，一次充值用四分之一个世纪 |
| 存在并发风险 | `productFiles` 是普通 ArrayList，两个线程同时 add 会互相覆盖，代码里自己标了 TODO |
| 性能显著提升 | P99 从 300ms 降到 80ms |

挣来的具体性有几种稳定来源，写作和改写时按这个清单找：

- **实测数字**：延迟、成本、行数、耗时，带单位
- **失败记录**：试了什么、怎么失败的。失败路径比成功路径更有信息量，也更没法编
- **算账**：把结论算出来给读者看，算式本身就是论证
- **代码坐标**：`BaseAgent.java:78-82` 这种行号引用，表明作者真读了
- **利益与动机分析**：谁从中获益、链条怎么走（"卖家可能同时赚差价和返利"）
- **误区纠正**："很多人以为是节点问题，反复换 VPN，其实是钱不够"——先摆常见错误归因，再给真原因

改写时的操作：把每个形容词判断（显著、极大、很快）替换成上表右列的某一种；替换不了的，要么降级为限定表述（"我没测过规模化场景，但从实现看……"），要么删。

## 机制解释

技术传播文的核心证据形态。"是什么"只值一分，"为什么非这样不可"才值十分。

检查每个技术描述是否停在表面：

- 表面（不合格）：PlanningTool 提供 create、update、mark_step、finish 四个命令。
- 机制（合格）：计划为什么要做成工具？因为放 prompt 里有三个死穴：格式靠正则碰运气、状态没人管、前端没法渲染进度。做成工具后，LLM 只能通过函数调用间接改状态，一致性由代码保证——这就是"不靠 LLM 自觉"的含义。

补机制的固定追问：为什么是这个方案而不是显而易见的那个？不这样做会发生什么？代价是什么？三问都有答案，这段才算有肉。

## 因果链查跳步

每个"所以/因此/这就是为什么"前面，检查"因为"给够了没有。AI 文常见的跳法：

- **相关当因果**："项目在 GAIA 榜单拿了 75.15%，说明这套设计有效"——榜单成绩和某个单点设计之间隔着无数变量，要么补上中间链条，要么把话说小（"至少说明这套组合在复杂任务上能打"）。
- **模糊归因**："业内普遍认为""专家指出"——清零。有具体出处就给出处，没有就改成自己的判断并承担它。
- **以复述代论证**：换个说法把结论再说一遍，冒充推理。删掉复述，读者不需要被说服两次。

## 类比纪律

好类比可以推演，坏类比只能装饰。

- 装饰类比（删）："Agent 引擎就像一位交响乐指挥家"——除了氛围什么都没提供。
- 可推演类比（留）："CountDownLatch 像一扇倒数门：设 3 个数，每个线程干完活减一，减到 0 门开"——读者可以顺着门的意象推出 await 的行为。

标准：类比引入后，后文能否用它继续解释新情况？不能就删。全文类比 ≤ 3 个，一个好类比反复用，胜过五个一次性类比。

## 代码克制

技术思想文里，代码是配角。规则：

- 一句话能说清的逻辑，不贴代码。"`stepPlan()` 把当前 in_progress 步骤标记完成、启动下一步"胜过贴 20 行循环。
- 必须贴时，只贴承载论证的关键行（≤ 15 行），删无关字段和样板，加行号坐标供读者回查。
- 代码块的存在测试：把它删掉，正文还读得通吗？读得通说明代码是装饰，删；读不通说明正文偷懒把解释外包给了代码，补正文。
- 长篇累牍贴代码是没消化的表现——作者没想清楚哪几行是关键，就全端给读者。

## 个人在场

第一人称的真实认知轨迹，既是人味的来源，也是一种论证——它展示了思考过程，让读者能验证你的推理。

三种可用形态：

- **预期修正**："我一开始以为 replan 每轮都会全量重排计划，看了代码才发现只是 mark_step 推进——比想象的保守。"
- **现场反应**："读到 `return "finish"` 靠字符串当退出信号，我愣了一下：这契约也太隐式了。"
- **承认边界**："多任务并发在生产里触发的概率我没有数据，这里只能给静态分析。"

铁律：必须真实。没有真实的认知轨迹就不写，宁缺毋滥——编造的"我试了"比没有人称更糟，一旦被识破，全文的信用清零。改写别人的稿子时，个人在场的素材只能来自作者（诊断阶段问），不能替作者编。

## 信息密度审计

最后一遍过滤：逐段问"删掉这段，读者损失什么？"

- 损失一个判断、一个机制、一个坑 → 留
- 只损失"完整性" → 删
- 说不清损失什么 → 肯定删

"正确但无用"是技术文最隐蔽的水分：那些每句话都对、但读者读完什么也带不走的段落。它们通常出现在"为了过渡""为了铺垫""为了严谨"的位置。

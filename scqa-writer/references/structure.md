# 结构层：骨架手术

结构是文章的骨架。AI 文的结构病只有一种，但表现形式很多：**提纲扩写**——先列一个"全面"的大纲，再往每个格子里平行填充。填得再好，节与节之间也没有论证关系，读者读完一节没有任何理由读下一节。

本文件是骨架手术的操作手册，按手术顺序排列。

## 第一步：逆向大纲

把文章还原成它的提纲，病灶自己会显形。

操作：逐段（或逐节）抽出它实际在说的论点，一行一条，排成列表。然后检查三件事：

1. **指向性**：每条论点是否都在支撑同一个中心判断？支撑不上的，就是为凑全面而写的。
2. **承接性**：相邻两条之间，能否写出"读完上一条，读者自然会问 X，下一条答 X"？写不出的地方就是抽屉式罗列的断点。
3. **同构性**：每条的形态是否雷同（都是"介绍一个组件"）？雷同说明是按对象组织，不是按问题组织。

诊断示例（真实病例）：一篇拆解 Agent 引擎的文章，逆向大纲是"BaseAgent → ReActAgent → 两种模式 → PlanningAgent → 并发 → ExecutorAgent → SummaryAgent → LLM 引擎 → 工具体系 → SSE → SOP → 总结"。十四节，每节介绍一个类。这是类图的散文版，不是文章——它回答的问题是"这个项目有什么"，而读者关心的是"这个项目哪里值得我学、哪里是坑"。

## 第二步：一句话判断

中心思想必须是判断句，不是名词短语。

- 主题（不合格）：JoyAgent 的 Agent 引擎实现原理
- 判断（合格）：把计划做成工具、让代码管状态，是这套引擎最值得抄的设计；而它的并发隔离是照抄前要先补的坑

判断句自带取舍：它规定了哪些内容是主角（计划工具、并发隐患），哪些只配一句话带过（SummaryAgent、SSE 心跳）。提炼不出判断句的文章，写多长都是资料汇编。

从原文提炼判断的三个入口：
- 作者在哪一节写得最起劲、细节最多？那里往往埋着真判断。
- 原文的"总结"节里哪条最不像套话？
- 问作者：如果读者只能记住一句话，你希望是哪句？

## 第三步：选题取舍

传播文不是覆盖文。判断一节该不该活下来，看它有没有以下任何一种价值：

- **反直觉**：和读者的默认预期相反（"计划不该放 prompt 里，该做成工具"）
- **有取舍**：展示了一个两难和作者的选择（"轻量的代价是并发安全"）
- **有坑**：踩过才知道的失败路径（"ArrayList 竞态怎么丢数据"）
- **可迁移**：读者能带走用到自己项目里的思想

四条都不沾、只是"这个模块也存在"的节：整节砍掉，或压成一句话并入相邻节。砍不下手时记住：读者的注意力是预算，你替他省的每一分钟都是文章的功德。"全面"是 AI 的病，"取舍"才是人的判断。

## 第四步：SCQA 开头

开头的任务是让读者在三段之内明白"这和我有什么关系、你要说什么"。SCQA 是最稳的模板：

- **S 情境**：读者已有的经验，一两句，建立共识。注意是"读者的经验"，不是"背景介绍"。
- **C 冲突**：情境里的真矛盾。必须具体到能让人皱眉，"随着 AI 的发展，Agent 越来越重要"不是冲突，是废话。
- **Q 疑问**：C 自然引出的问题，通常隐含，不必明写。
- **A 答案**：你的判断，提前亮出来。不要卖关子留到结尾。

示例：

> 让 LLM 做一份"美元黄金走势分析报告"，光靠聊天不够——它得查数据、画图、生成报告，也就是反复"思考→动手→看结果"。（S：读者知道 LLM 聊天，也隐约知道 Agent）
> 但谁来管这个循环？什么时候停？LLM 说"我要建个计划"，计划放哪、状态谁维护？大多数框架把计划塞在 prompt 里靠解析文本维持，一改就碎。（C：具体到工程细节的矛盾）
> JoyAgent 给了个反直觉的答案：把计划本身做成一个工具，状态机交给 Java 代码管。这篇讲清这个设计为什么聪明，以及它在并发上留了什么坑。（A：判断先行，顺带预告取舍）

变体：高管摘要式用 A→S→C（结论劈头就来），危机式用 C→S→A（先抛问题）。技术博客默认 S→C→A。

## 第五步：疑问链重排

好文章的节与节是追问关系：每节收尾时，读者心里自然升起下一个问题，下一节正好回答它。这是金字塔原理"疑问-回答"式展开的实际用法。

操作：给新骨架的每一节写两行元信息（只在骨架阶段用，成稿删掉）——

```
节标题：把计划做成工具
回答的问题：计划不放 prompt 里，那放哪？（由上一节"prompt 维护计划会碎"引出）
```

任何一节写不出"由上一节引出"，说明链条断了：要么调节的顺序，要么在上一节结尾补一句引出问题的钩子，要么这节根本不属于这条链（考虑砍）。

## 第六步：标题

标题写读者的问题或利益，不写对象名：

- 对象名（弱）：充值流程 / PlanningAgent 详解 / 并发设计
- 问题与利益（强）：充值避坑：99% 的人会卡在这一步 / 把计划做成工具：最值得抄的一个设计 / 并发快是快，但你的文件可能悄悄丢

节标题里能放 C（冲突）就放 C——冲突前置是最便宜的钩子。全文标题 ≤ 20 字，禁对仗、禁悬念党（"结果让人意外"）、禁 SEO 腔（"一文读懂""最全指南"——除非内容真配得上）。

## 结构层自检

- [ ] 逆向大纲每条都指向中心判断
- [ ] 相邻节之间写得出"上节引出的问题"
- [ ] 没有纯覆盖节
- [ ] 开头三段内出现冲突和判断
- [ ] 标题面向读者问题
- [ ] 结尾停在判断/行动/后果，不复述、不展望

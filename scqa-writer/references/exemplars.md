# 正样本拆解：好文章好在哪

黑名单只能告诉你什么不能写，这个文件告诉你好的长什么样。两个真实样本：一篇范文，一篇病例，都来自本 skill 校准时的实际分析。

## 范文：《giffgaff 全网最全使用指南》（X @noahduck283）

一篇工具类长文，读起来毫无 AI 味。值得抄的六个手法：

### 1. 按读者路径组织，不按知识分类

目录是"拿到卡 → 用好卡 → 玩转卡"，每一节对应读者旅程里的一个节点，而不是"产品介绍 / 功能列表 / 常见问题"这种作者视角的分类。技术文的对应做法：按"读者从不懂到会用这个思想"的路径排节，不按代码目录结构排。

### 2. 标题全是冲突前置

"充值避坑：99% 的人会在这一步卡住""这个默认设置可能悄悄扣你的钱""用了这么久可能一直填错了"——每个标题都是 SCQA 里的 C，读者扫一眼目录就被至少三个标题勾住。对象名标题（"充值流程""语音信箱设置"）做不到这一点。

### 3. 失败记录当证据

"招行 Visa 信用卡多次尝试，全部失败。换美国节点、换英国节点，照样不行。"三个短句，信息密度极高，而且没法编——这就是挣来的具体性。更进一步，它先摆常见的错误归因再纠正："很多人以为是节点问题，反复换 VPN，其实问题就是钱不够。"误区纠正是最有人味的论证形态之一。

### 4. 算账代替形容

"£15 ÷ £0.6/年 = 25 年。一次充值，够用四分之一个世纪。"不说"成本极低"，把账算给你看。三种保号方案各算一遍，结论自己从算式里长出来。技术文的对应做法：延迟、token 成本、代码行数，能算就算。

### 5. 有立场，替读者做判断

"直接说结论：两者根本不是同一个量级。""不要浪费时间反复试了。""这不是说所有卖卡的人都不能买。但你要知道自己到底在为哪部分付钱。"最后这句还揭了利益链（卖家同时赚差价和 £5 返利）——作者站在读者一边，这种立场感是中立百科腔永远给不了的。

### 6. 节奏靠单句成段

"所以别把它当流量卡。""在国内，它的正确姿势是接码和保号。"关键判断独立成段，前后是厚段。长短交错不是修辞技巧，是在给读者的注意力打拍子。

### 重要校准

这篇范文大量使用粗体标题列表、数字步骤、"一句话总结"——全是去 AI 味黑名单上的"痕迹"。它依然是好文章，因为每一处形式都装着真信息。**痕迹清单抓的是症状，病原是内容空洞。内容挣来了，形式怎么方便读者怎么来。**

## 病例：《Agent 引擎实现原理》（本 skill 作者旧稿）

一篇下了真功夫的源码解析——有行号、有机制解释、有坑的分析——但读起来仍是文档不是博客。病灶和范文逐条对照：

| 维度     | 病例                                        | 范文                           |
| -------- | ------------------------------------------- | ------------------------------ |
| 组织方式 | 按类图排十四节，每节一个组件                | 按读者路径排                   |
| 标题     | "BaseAgent——把循环和状态管起来"（对象名） | "99% 的人会卡在这一步"（冲突） |
| 取舍     | 穷尽覆盖，SummaryAgent、SSE 心跳都单独成节  | 只写读者用得上的，其余不提     |
| 人称     | 全文无"我"                                  | 处处是作者的实测和立场         |
| 形态     | 40+ 个问答卡片，节节同构                    | 叙述为主，形式跟着信息走       |

值得注意的是它的论证层其实不差：并发隐患一节把 ArrayList 竞态的四步交错写得清清楚楚，还有代码自带的 TODO 注释当铁证——这正是挣来的具体性。问题在结构层（无取舍、无疑问链）和人称（作者不在场）。这个病例说明：**语言层痕迹不重的文章照样可以 AI 味很重，humanizer 类工具对它无效，必须动骨架。**

## 改写示范

**改写前（原稿开头式）：**

> 本文将深入剖析 genie-backend 的 Agent 编排引擎，全面介绍其四级类继承体系、两种执行模式以及并发设计，帮助读者彻底搞懂 Agent 引擎的实现原理。

病：报账式开头（本文将）、讲义腔（深入剖析、彻底搞懂）、读者许诺、无冲突无判断。

**改写后：**

> 让 LLM 写一份走势分析报告，它得查数据、画图、生成网页，反复"想一步、做一步"。谁来管这个循环？大多数框架把计划塞在 prompt 里靠正则解析维持，一改就碎。JoyAgent 的方案反直觉：把计划本身做成一个工具，状态机交给 Java 代码管。这个设计值得抄，但抄之前得看清它在并发上留的三个坑。

S（读者知道 LLM 要动手）→ C（计划放 prompt 里会碎）→ A（做成工具 + 有坑要看清），判断和取舍在开头就立住了。

**改写前（知识卡片式）：**

> **CountDownLatch 是什么**
> Java 的并发工具，像一扇「倒数门」。设 3 个数，3 个线程各干完一个活就 countDown 减一，主线程在门这儿 await 等着。

**改写后（熔进叙述）：**

> 三个子任务各跑各的，主线程用 CountDownLatch 等它们——像一扇倒数门，每个线程干完活减一，减到零门开，主线程再合并结果。

同样的类比，从卡片熔回正文，读者的阅读流不被打断。类比本身是好的（可推演），要改的只是它的存在形态。
